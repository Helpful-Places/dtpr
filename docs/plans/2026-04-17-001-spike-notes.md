# Unit 0.5 Spike Notes

Three architectural commitments resolved before starting Unit 1.

## TL;DR

**Adopt Path B (Vue SSR in Worker).** `@vue/server-renderer`'s `renderToString` works in `workerd` with negligible overhead. The bundle cost is modest, and collapsing two renderers into one eliminates Unit 5's parallel-implementation burden and reduces Unit 7 equivalence testing to a trivial smoke check.

**Adopt `@modelcontextprotocol/ext-apps`.** `registerAppTool` + `registerAppResource` bundle cleanly under `nodejs_compat` and produce the expected MCP Apps protocol responses end-to-end. The `App` wrapper composes cleanly with `@hono/mcp`'s `StreamableHTTPTransport`.

**Toolchain (Vite library mode + `vite-plugin-dts` + `@vitejs/plugin-vue`) works.** A real SFC with `<script setup lang="ts">` plus a `/core` TS entry produced correct, complete `.d.ts` on the first build.

## 1. Renderer architecture â€” Path A vs Path B

### Measurement

Built a spike Worker at `api/spike/worker.ts` that imports `vue` + `@vue/server-renderer`, renders a 10-section Ă— 3-elements-per-section fixture via `renderToString`, exposes `/` and `/bench?n=100` endpoints.

| Variant | Total upload | Gzipped |
|---|---|---|
| Baseline `@dtpr/api` Worker (today, no MCP, no Vue) | 62 KiB | 15 KiB |
| Baseline + Vue SSR only | 277 KiB | 65 KiB |
| Baseline + Vue SSR + MCP SDK + ext-apps + Hono + zod (full Unit 6 bundle) | 1521 KiB | 278 KiB |

**Vue SSR delta over baseline**: +215 KiB raw / **+50 KiB gzipped**. This sits well under the plan's 200 KiB decision threshold.

**Render latency** (100 iterations, 30-element fixture, wrangler dev local with workerd):

- **p50**: 1 ms
- **p99**: 10 ms

Decision threshold was p99 < 50 ms. Measured p99 is 5Ă— under. Cold-start HTTP round-trip was 12 ms on a fresh Worker.

### Verdict

**Path B (Vue SSR in Worker) is adopted.** All three thresholds satisfied with margin. The architectural consequence:

- `@dtpr/ui/html`'s `renderDatachainDocument` becomes a ~50-line wrapper that calls `renderToString` on `@dtpr/ui/vue`'s `<DtprDatachain>` and wraps the body in the `<!doctype html>` shell + inline `<style>` + inline `<script>`.
- Unit 5 shrinks substantially â€” no parallel fragment renderers, no separate `category-section.ts`/`element-detail.ts`/etc. string-template functions. One implementation per component (the Vue SFC) feeds both renderers.
- Unit 7 (equivalence testing) shrinks to a smoke test: "does `renderToString(<DtprDatachain>)` output wrap cleanly in the document shell and run the vanilla JS accordion?" No cross-renderer diff harness needed.
- Vue SFCs must use **unscoped CSS only** (no `<style scoped>`) so `renderToString` doesn't emit `data-v-*` hydration attributes in the Worker path. Enforced in Unit 4.

### Fallback (unused)

Path A (parallel renderers) remains documented in the plan as a fallback if a future Worker limit change or Vue regression invalidates the spike result. Rollback is mechanical.

## 2. MCP Apps SDK â€” ext-apps vs native `registerResource`

### Measurement

Extended the spike Worker to import `registerAppTool` + `registerAppResource` from `@modelcontextprotocol/ext-apps/server`. Created an `McpServer`, registered `render_datachain` tool and `ui://dtpr/datachain/view.html` resource, mounted on `/mcp` via `@hono/mcp`'s `StreamableHTTPTransport`.

End-to-end verification:

- **`initialize`** â†’ server returns `protocolVersion: "2025-06-18"`, capabilities include `resources.listChanged` + `tools.listChanged`.
- **`tools/list`** â†’ `render_datachain` appears with `_meta.ui.resourceUri: "ui://dtpr/datachain/view.html"` (and deprecated `_meta["ui/resourceUri"]` mirror). Input schema properly translated from Zod to JSON Schema.
- **`resources/list`** â†’ returns the UI resource with `mimeType: "text/html;profile=mcp-app"` as the MCP Apps spec requires.
- **`resources/read`** â†’ returns the full rendered HTML document, Vue-SSR-produced body inside the doctype shell. 4.7 KB of content for the 10-section fixture.

No workerd runtime failures. No missing Node APIs. Bundle composed correctly with `nodejs_compat`.

### Verdict

**Adopt `@modelcontextprotocol/ext-apps`.** The library's `registerAppTool` / `registerAppResource` produce the right wire format without hand-rolling `_meta.ui.*` fields. `App` wrapper integration is not needed for server-only registration â€” the helpers operate directly on `McpServer`.

### Implementation note for Unit 6

The actual signatures (from `node_modules/@modelcontextprotocol/ext-apps/dist/src/server/index.d.ts`):

```
registerAppResource(
  server: Pick<McpServer, "registerResource">,
  name: string,
  uri: string,
  config: McpUiAppResourceConfig,
  readCallback: McpUiReadResourceCallback,
): RegisteredResource

registerAppTool(
  server, name, toolConfig, callback,
): RegisteredTool
```

The stable-URI-with-state pattern works: register one URI (`ui://dtpr/datachain/view.html`) at server setup; the tool handler mutates rendering state; the resource's `readCallback` reads that state at `resources/read` time. For production, per-session state isolation is needed (the spike used a module-level slot). See Unit 6 implementation notes.

## 3. Toolchain â€” Vite library mode + vite-plugin-dts + `<script setup lang="ts">`

### Measurement

Scaffolded a minimal `packages/ui/` package with:

- `src/core/index.ts` â€” one real function (`extract`) plus an interface (`LocaleValue`) and a constant (`HEXAGON_FALLBACK_DATA_URI`)
- `src/vue/DtprIcon.vue` â€” real `<script setup lang="ts">` SFC with typed props (`src?`, `alt`, `size?`) + defaults
- `vite.config.ts` with two entries (`core/index`, `vue/index`), `@vitejs/plugin-vue`, `vite-plugin-dts`, `cssCodeSplit: false`, `external: ['vue']`

`pnpm --filter @dtpr/ui build` produced:

```
dist/core/index.d.ts     â€” full extract() signature + LocaleValue interface
dist/core/index.js       â€” 520 bytes
dist/vue/index.d.ts      â€” re-exports DtprIcon + LocaleValue
dist/vue/DtprIcon.vue.d.ts â€” full DefineComponent type with Props and default resolution
dist/vue/index.js        â€” 603 bytes (Vue externalized)
dist/style.css           â€” single compiled stylesheet (30 bytes for the stub)
```

Build completed in 636 ms.

### Verdict

**Toolchain is solid.** Key wins:

- `vite-plugin-dts` correctly extracts prop types from `<script setup lang="ts">` including `withDefaults` â€” no incomplete type emission.
- Multi-entry library mode with per-entry subdirectories in `dist/` works cleanly for the `@dtpr/ui/core` + `@dtpr/ui/vue` subpath exports map.
- `external: ['vue']` keeps Vue out of both bundles, aligning with `peerDependencies: { vue: ^3.5 }`.
- Single-file `styles.css` (via `cssCodeSplit: false`) matches R16.

### Notes for Unit 1 scaffolding

- Keep the same `vite.config.ts` shape. Add `html/index` as a third entry once Unit 5 lands (or, if Path B collapses `/html`, drop `/html` from the exports map entirely and inline the document-shell helper into `/vue`).
- Add `@dtpr/api/schema` as a workspace dep in Unit 2 â€” Unit 0.5 scaffolded only Vue + TS, not the schema import. That specific resolution path (workspace dep importing a tsup-built `.d.ts`) is untested in this spike. **Recommend verifying it in Unit 2 before closing that unit**, by having one type in `src/core/types.ts` import from `@dtpr/api/schema` and re-export.
- `tsconfig.json` currently has `noEmit: true` â€” `vite-plugin-dts` handles declaration emission separately via `vue-tsc`, so that's fine. No separate `tsconfig.build.json` needed unless we later drop `vite-plugin-dts`.
