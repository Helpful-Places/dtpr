# `dtpr-element-design`

The element-tier schema skill. Scope is **one proposed element** (add, edit, or retire). Output is a YAML fragment skeleton matching the canonical element shape plus a symbol direction as prose (not SVG). Locale coverage is skeleton only — `title` and `description` are drafted in English across all six locales, and actual translation is out of scope.

## When to fire

- The user proposes a new element for a specific disclosure concept.
- The user wants to retire an existing element and replace it with something more precise.
- The user asks "how would DTPR describe X" as a prelude to drafting.
- The user pastes YAML and wants a cleaned-up element skeleton with proper locale structure.

## Trigger phrases

"propose a new element", "how would DTPR describe X", "draft an element for Y", "retire the cloud_storage element", "replace the X element with something better", "brainstorm a new element for LLM hallucination".

## Workflow

**Phase 0 — Accept the concept and target category.** When multiple categories could fit, surfaces the ambiguity and asks the user to choose.

**Phase 1 — Collision check.** Calls [`get_element`](/mcp/tools/get-element) on the candidate id and [`list_elements`](/mcp/tools/list-elements) with `query=<candidate_title>` to surface near-duplicates. Flags collisions for user disambiguation.

**Phase 2 — Corpus lookup.** Tags: `element:<id>` for prior research on the element, `category:<id>` for context, `concept:<slug>`, `framework:<name>`, `standard:<name>`. Dispatches a researcher via `Task` on miss if available.

**Phase 3 — Draft the YAML fragment.** Matches the canonical shape of `api/schemas/ai/<version>/elements/<id>.yaml`: `id`, `category_id`, six-locale `title` and `description`, `citation`, `symbol_id`, optional `variables`.

**Phase 4 — Symbol direction.** A **prose description** suitable for a designer — posture, object, motion — with rationale for sign-scale legibility. Does not emit SVG. Notes reuse when an existing symbol applies.

**Phase 5 — Inline the Comprehension check** block.

**Phase 6 — Emit the `schema:new` handoff** and recommend a sibling when relevant (e.g., `dtpr-category-audit` if the proposed element exposes category overlap).

## Output

A proposal with Scenario / Proposed changes {Add|Edit|Retire} / Symbol direction / Comprehension check / Next step (`schema:new` command line) and a sibling recommendation when relevant.

## Non-goals

Symbol output is a **direction**, not an SVG. Translation is out of scope. Does not modify files under `api/schemas/` or run `schema:new`. Does not audit a category's coherence — route to [`dtpr-category-audit`](/plugin/skills/category-audit). Does not critique the datachain-type shape — route to [`dtpr-datachain-structure`](/plugin/skills/datachain-structure). Does not describe a system — route to [`dtpr-describe-system`](/plugin/skills/describe-system).
